What it does

Shelf lets users track books they're reading, maintain reading streaks, get AI-generated summaries, and discover new books through friends' activity. Built as a Next.js app with Supabase for auth and storage, and Claude for generating book summaries and personalized recommendations.

Key Features
Core

Book Tracking

Search Open Library API, add to shelf, track reading progress with page counts and status (reading, finished, abandoned).

Core

Reading Streaks

Daily check-in system. Log minutes read to maintain streak. Streak freeze available once per week.

AI

Smart Summaries

Claude generates chapter summaries, key takeaways, and discussion prompts for any book on the shelf.

Social

Friend Activity

Follow friends, see what they're reading, react to their reviews. Privacy-first: all sharing is opt-in.

System Architecture
Technology Choices

Why Next.js App Router

Server components for book detail pages (SEO-friendly). API routes co-located with the app (same-origin, no CORS). Streaming for AI summary generation.

Why Supabase

Auth out of the box. Row Level Security means the API doesn't need manual permission checks. Realtime subscriptions for friend activity feed updates.

Why Claude for Summaries

Long context window handles full book chapters. Structured output for consistent summary format. Low hallucination rate on factual book content.

Why Open Library

Free, no API key required. Covers 20M+ books. ISBN lookup + full-text search. Rate limits are generous (100 req/min).

Adding a Book to the Shelf
01
User searches for a book
SearchModal sends query to /api/search, which proxies to Open Library. Results cached for 5 minutes.
02
User clicks "Add to Shelf"
POST to /api/books with Open Library work ID, title, author, cover URL. Server creates row in books table with status "reading".
03
Dashboard updates optimistically
BookCard appears immediately via optimistic update. Supabase Realtime confirms the insert. If insert fails, card rolls back with toast error.
04
Summary generation triggered on first view
When user opens book detail, /api/books/[id]/summary checks if a summary exists. If not, fires a background request to Claude. Summary streams back and is persisted to summaries table.
05
Friend feed picks it up
If the user has sharing enabled, the book addition triggers a Supabase Realtime event. Friends following this user see the new book in their activity feed within seconds.
Streak Check-in Flow
01
User logs reading time
StreakCounter component sends POST to /api/streaks with minutes read. Server validates (minimum 5 minutes to count).
02
Server checks streak continuity
Queries last check-in date. If yesterday: increment streak. If today: update minutes. If older: check for freeze token, reset if none.
03
Response includes streak state
Returns current streak count, longest streak, freeze availability, and whether today's check-in is complete. StreakCounter animates the flame icon.

What We Did Right

Same-origin API routes

All API routes live inside the Next.js app. No separate backend service, no CORS configuration, no extra deployment target. This keeps the architecture simple and the deploy fast.

Lazy summary generation

AI summaries are generated on first view, not on book add. This avoids wasting Claude API credits on books the user might never open, and it means the "add book" flow stays instant.

Privacy-first social features

Every book defaults to private. Users must explicitly opt in to share each book. The is_public column on both books and reviews tables enforced via RLS.

What Needs Fixing

HIGH

The /api/books/[id]/summary endpoint calls Claude with no rate limiting. A user could hammer this endpoint and rack up API costs. Add per-user rate limiting (max 5 summaries per hour).

In src/app/api/books/[id]/summary/route.ts, add rate limiting before the Claude call. Query the summaries table for count of rows created by this user in the last hour. If >= 5, return 429 with { error: "Rate limit exceeded. Max 5 summaries per hour." }. Use the authenticated user ID from Supabase auth.
MEDIUM

If a user checks in at 11:55 PM and then at 12:05 AM the next day, the streak logic sees it as the same day (no credit for day 2). Consider using a rolling 24-hour window instead of calendar days, or allow a 2-hour grace period past midnight.

In src/lib/streaks.ts, update the streak continuity check. Instead of comparing calendar dates, check if the gap between last_checkin and now is less than 26 hours. This gives a 2-hour grace period past midnight for users in edge timezones. Add a test case for the 11:55 PM -> 12:05 AM scenario.
LOW

The search proxy passes Open Library results directly to the client. If the external API returns unexpected fields or malformed data, the BookCard component could crash. Add a Zod schema to validate and sanitize the response.

In src/lib/openlib.ts, add a Zod schema for the Open Library search response. Validate title (string), author_name (string[]), cover_i (number, optional), and key (string). Use .safeParse() and filter out any results that fail validation. This prevents malformed data from crashing BookCard.
